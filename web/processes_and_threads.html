1. Processes

What they are, what they own, and how the OS manages them.

Anatomy of a Process

Process Virtual address space Isolation

A process is the operating system's unit of resource ownership and protection. When you launch a program, the OS creates a process that encapsulates everything the program needs to run: its own virtual address space, one or more threads of execution, open file descriptors, security credentials, and environment variables. Two processes cannot accidentally corrupt each other's memory because the hardware (via the MMU and page tables) enforces address-space isolation. A pointer in process A simply has no meaning inside process B's address space.

Every process is identified by a unique process ID (PID). It also carries metadata the OS needs for scheduling, accounting, and security: the user who owns it, the priority, CPU time consumed, the current working directory, and signal masks. Conceptually, a process is a running instance of a program combined with all the kernel-managed state that instance needs.

Each process has its own virtual address space. The MMU prevents cross-process memory access.

Example: When you open two tabs in a modern browser like Chrome, each tab often runs in a separate process. If a JavaScript bug causes a crash in one tab, the other tab survives because the OS isolates their memory. This is the fundamental value of process-level isolation.

The cost of this isolation is overhead. Creating a process means allocating a new page table, copying or setting up file descriptor tables, and initializing kernel structures. Context-switching between processes requires flushing or tagging TLB entries because the virtual-to-physical mapping changes entirely. These costs are not catastrophic on modern hardware, but they are real, and they motivate the existence of threads as a lighter alternative for concurrency within a single program.

Lifecycle and States

States Scheduling

A process moves through a well-defined set of states during its lifetime. The details vary slightly between OSes, but the core model is universal. A process begins in a created (or new) state while the OS sets up its structures. It then moves to ready, meaning it is prepared to run but waiting for the scheduler to assign it a CPU core. When the scheduler picks it, the process enters the running state. If the process issues a blocking system call (such as reading from disk or waiting for a network packet), it moves to blocked (waiting). When the I/O completes, it returns to ready. Eventually, when the process calls exit() or is killed, it enters a terminated (zombie on Linux) state until its parent collects its exit status.

Example: A web server process calls read() on a socket. It has no data yet, so the kernel moves it to Blocked. When a client sends a request, the NIC raises an interrupt, the kernel copies data into the buffer, and the process moves back to Ready. The scheduler eventually runs it again.

2. Threads (Kernel Threads)

Concurrency within a single process, scheduled by the OS kernel.

Thread vs. Process

Thread Concurrency Process

A thread is the OS's unit of CPU scheduling. A process always has at least one thread (the main thread). Additional threads can be created within the same process, and they all share the same virtual address space, file descriptors, and other process-wide resources. What each thread owns privately is a separate stack, a set of CPU registers (including its own program counter), and thread-local storage.

This sharing is what makes threads cheaper than processes. Creating a second thread does not require a new page table or a new set of file descriptors. Context-switching between two threads of the same process does not require a TLB flush because the virtual-to-physical mapping stays the same. The kernel just saves one thread's register state and restores another's. The trade-off is that threads within a process are not isolated from each other. A bug in one thread can corrupt memory visible to all threads in that process. 

  Process (PID 5500)
 ┌────────────────────────────────────────────────────┐
 │  Shared: code, data, heap, file descriptors, PID   │
 │                                                     │
 │  Thread 1            Thread 2            Thread 3   │
 │ ┌──────────┐       ┌──────────┐       ┌──────────┐ │
 │ │ Stack    │       │ Stack    │       │ Stack    │ │
 │ │ Registers│       │ Registers│       │ Registers│ │
 │ │ PC       │       │ PC       │       │ PC       │ │
 │ │ TLS      │       │ TLS      │       │ TLS      │ │
 │ └──────────┘       └──────────┘       └──────────┘ │
 └────────────────────────────────────────────────────┘

Three threads inside one process share address space but have private stacks, registers, and TLS.

Example: A Java application server handles incoming HTTP requests. Each request is dispatched to a thread from a thread pool. All threads share the same JVM heap (where cached data and connection pools live), but each thread has its own call stack. This avoids the enormous overhead of forking a new process per request.

The shared heap and shared globals are both the strength and the danger of multithreading. They allow efficient data sharing without copies. But they also mean that any thread can write to any shared location at any time. Without explicit synchronization, this leads to data races, torn reads, and heisenbugs that are notoriously difficult to reproduce and diagnose. That is why thread synchronization primitives exist, and we cover them in the IPC section below.

3. Lightweight / User-Level Threads

Threads managed in user space, multiplexed onto fewer kernel threads.

Threading Models: N:1, 1:1, and M:N

Green threads Fibers Coroutines

The term "lightweight thread" refers to a thread whose creation, destruction, and scheduling are managed primarily in user space rather than by the kernel. The idea is simple: kernel thread operations involve system calls, which involve a privilege-level transition (user mode to kernel mode), which is expensive relative to a plain function call. If we can schedule thousands of logical tasks without making thousands of system calls, we gain efficiency.

There are three classical models for mapping user-level threads (U) to kernel threads (K): 

  N:1 (Many-to-One)          1:1 (One-to-One)          M:N (Many-to-Many)

  U1 U2 U3 ... Un            U1  U2  U3  Un            U1 U2 U3 U4 U5 U6
   \  |  |  /                 |   |   |   |              \  |  /  \  |  /
    \ | | /                   |   |   |   |               \ | /    \ | /
  ┌─────────┐             ┌───┐┌───┐┌───┐┌───┐          ┌─────┐  ┌─────┐
  │  K1     │             │K1 ││K2 ││K3 ││Kn │          │ K1  │  │ K2  │
  └─────────┘             └───┘└───┘└───┘└───┘          └─────┘  └─────┘

  - All user threads on      - Each user thread       - User-level scheduler
    one kernel thread           maps to one kernel       multiplexes M user
  - No true parallelism         thread                   threads onto N kernel
  - One blocks → all block    - True parallelism         threads
                              - Higher kernel overhead  - Best of both worlds
                                                        - Most complex to build

The three classical user-to-kernel thread mapping models.

N:1 was used by early Java (green threads before JDK 1.2) and early Ruby. It is simple, but one blocking system call blocks all user threads because the single kernel thread is stuck. 1:1 is what Linux pthreads and Windows threads use today: each user thread is backed by a kernel thread, giving true parallelism on multicore CPUs. The cost is that creating 100,000 threads means 100,000 kernel objects and 100,000 stacks, which may use gigabytes of memory. M:N multiplexes many user-level threads onto a smaller pool of kernel threads, offering parallelism without the per-thread kernel overhead. It is the model used by Go goroutines, Erlang processes, and Java's Project Loom virtual threads.

Example: A Go web server can handle 500,000 concurrent goroutines. Each goroutine has a tiny initial stack (a few KB, grown on demand). The Go runtime schedules these goroutines onto a pool of OS threads equal to GOMAXPROCS (typically the number of CPU cores). This M:N model is why Go excels at massively concurrent network services.

Modern Green-Thread Runtimes

Go goroutines Java virtual threads Erlang processes Rust async/await

Modern runtimes refine the M:N model with cooperative or hybrid scheduling. In Go, goroutines yield at certain safe points (function calls, channel operations, system calls) so the runtime's scheduler can switch between them without a system call. When a goroutine would block on I/O, the runtime parks it and reuses the underlying OS thread for another goroutine. This means blocking I/O does not waste an OS thread.

Java's Project Loom (production since JDK 21) introduces virtual threads. A virtual thread looks like a normal java.lang.Thread but is backed by a continuation that the JVM can mount and unmount from carrier (platform) threads. You can create millions of virtual threads. When one blocks on socket.read(), the JVM unmounts its continuation from the carrier thread and schedules another virtual thread onto it. The programming model is synchronous blocking code, but the runtime achieves the efficiency of asynchronous I/O. 

  Java Virtual Threads (Project Loom):

  Virtual Thread 1  ─┐
  Virtual Thread 2  ─┤  mounted/unmounted by JVM scheduler
  Virtual Thread 3  ─┤
      ...            │
  Virtual Thread N  ─┘
                      ▼
           ┌──────────────────┐
           │ Carrier Thread   │  (small pool of OS threads)
           │ Pool (ForkJoin)  │
           └──────────────────┘

  When VT blocks on I/O → unmounted, carrier freed for another VT
  When I/O completes   → VT re-mounted on any available carrier

Virtual threads are scheduled by the JVM, not the OS. Blocking I/O does not waste an OS thread.

Erlang takes a different extreme. Every Erlang "process" is a lightweight user-space entity with its own tiny heap. Erlang processes do not share memory at all. They communicate exclusively by message passing. The BEAM VM schedules these processes across OS threads with preemptive reduction counting (each process gets a fixed number of reductions before being preempted). This model trades shared-memory performance for absolute isolation and fault tolerance, which is why Erlang dominates telecom and messaging systems.

Rust's async/await model is yet another approach. Rust futures are state machines compiled by the compiler. An async runtime like Tokio polls these futures on a thread pool. There are no green thread stacks at all. Each future is an enum that stores only the state needed at each .await point. This is extremely memory efficient but requires the programmer to use async annotations pervasively.

4. Inter-Process & Inter-Thread Communication

How isolated processes and co-resident threads exchange data and coordinate.

IPC Mechanisms

Pipes Shared memory Sockets Message queues Signals

Because processes have isolated address spaces, they need explicit OS-mediated mechanisms to communicate. There is no single best IPC mechanism. Each represents a different trade-off between latency, throughput, complexity, and whether the communicating processes are on the same machine.

Pipes are the simplest IPC mechanism. A pipe is a unidirectional byte stream: one process writes, another reads. Named pipes (FIFOs) allow unrelated processes to communicate via a filesystem path. Pipes are excellent for producer-consumer patterns and are the backbone of the Unix shell pipeline model (ls | grep foo | wc -l).

Shared memory is the fastest IPC mechanism because it eliminates copying. Two processes map the same physical page into their respective virtual address spaces. Once mapped, reads and writes appear as normal memory operations with no system call overhead. The danger is that shared memory requires explicit synchronization (semaphores or mutexes) to avoid races. It is the mechanism of choice for high-performance systems like database buffer pools and real-time audio processing.

  Process A                              Process B
 ┌────────────────┐                     ┌────────────────┐
 │ Virtual page 7  │────┐          ┌────│ Virtual page 3  │
 └────────────────┘     │          │    └────────────────┘
                        ▼          ▼
                   ┌────────────────────┐
                   │  Physical frame 42 │   ← same physical memory
                   └────────────────────┘

  A writes to its virtual page 7 → B sees the change at its virtual page 3
  No copies, no system calls for the data transfer itself
  But: synchronization (mutex, semaphore) is required

Shared memory maps the same physical frame into two processes' virtual address spaces.

Message queues (POSIX mq_* or System V msgget/msgsnd) provide structured message passing through the kernel. They add framing (each message has a boundary and optionally a type) at the cost of kernel copies. They are useful when you want decoupled, typed communication without managing raw byte streams.

Sockets generalize pipes to bidirectional communication and work across machines. Unix domain sockets provide fast local IPC (no network stack overhead), while TCP/UDP sockets enable network-transparent communication. Almost all modern distributed systems communicate via sockets.

Signals are asynchronous notifications delivered to a process. They are not a data-transfer mechanism. A signal tells a process that an event happened (e.g., SIGTERM for termination, SIGCHLD when a child exits). Signal handling is tricky because signals can arrive at any point during execution, making reentrancy a concern.

Example: PostgreSQL uses shared memory for its buffer pool (so all backend processes can access cached pages) and uses Unix domain sockets for client-server communication. It uses signals internally for coordination (e.g., SIGHUP to reload configuration). This combination is typical of production database systems.

Thread Synchronization Primitives

Mutex Condition variable Semaphore Read-write lock Atomic operations

Threads within a process share memory, so they do not need IPC. But they do need synchronization to avoid data races. The fundamental primitives are:

Mutex (mutual exclusion lock) — ensures only one thread can enter a critical section at a time. A thread that tries to lock an already-locked mutex blocks until the owner releases it. This is the workhorse of thread-safe programming.

Condition variable — allows a thread to block until some condition becomes true, without busy-waiting. A thread locks the mutex, checks the condition, and if it is false, calls wait(), which atomically releases the mutex and puts the thread to sleep. Another thread signals the condition variable when it makes the condition true. 

  Producer-Consumer with mutex + condition variable:

  Producer:                         Consumer:
    lock(mutex)                       lock(mutex)
    buffer.push(item)                 while buffer.empty():
    signal(not_empty)                     wait(not_empty, mutex)  ← releases mutex, sleeps
    unlock(mutex)                     item = buffer.pop()
                                      unlock(mutex)
                                      process(item)

The classic bounded-buffer pattern using a mutex and a condition variable.

Semaphore — a generalized counter. A binary semaphore acts like a mutex. A counting semaphore limits the number of threads that can enter a section simultaneously (e.g., limiting to N concurrent database connections). Read-write locks allow multiple concurrent readers but exclusive writers. Atomic operations (compare-and-swap, fetch-and-add) enable lock-free data structures for the highest-performance scenarios, but they are significantly harder to reason about correctly.

Example: A connection pool in a web server uses a counting semaphore initialized to the pool size (say 20). Each thread that needs a connection calls sem_wait(), decrementing the counter. If the counter reaches zero, subsequent threads block until a connection is returned via sem_post(). This naturally throttles concurrency to available resources.

5. Windows vs. Linux Process Models

Concrete implementation differences between the two dominant OS families.

Process Creation: fork() vs. CreateProcess()

Linux Windows Process creation

Linux and Windows take fundamentally different approaches to creating a new process.

On Linux (and all POSIX systems), the model is fork + exec. fork() creates an exact copy of the calling process: same code, same data, same open files. The child is a clone of the parent. The child then typically calls exec() to replace its own memory image with a new program. This two-step model is elegant and composable. Between the fork and the exec, the child can set up redirections, close unwanted file descriptors, change directories, or drop privileges. Modern Linux implements fork() with copy-on-write (COW): the parent and child initially share the same physical pages, and pages are duplicated only when either process writes to them. This makes fork very cheap if the child calls exec immediately.

Windows has no fork(). Instead, it uses CreateProcess(), which creates a new process and loads a program into it in a single call. All parameters (the executable path, command-line arguments, environment, security attributes, I/O handles to inherit) must be specified upfront. This is more explicit but less composable. The Windows model avoids the conceptual complexity of a forked child that has a copy of the parent's entire state (including locks, threads, and memory-mapped regions), which can be dangerous in multithreaded programs. 

  Linux:                                  Windows:

  parent                                  parent
    │                                       │
    ├─ fork() → child (clone of parent)     ├─ CreateProcess("app.exe", ...)
    │             │                         │        │
    │             ├─ setup (redirects,      │        └─ new process with
    │             │   close fds, chdir)     │           app.exe loaded
    │             │                         │           (no intermediate
    │             └─ exec("app")            │            clone state)
    │                  │                    │
    │               app running             │       app.exe running
    ▼                                       ▼

Linux separates cloning from program loading. Windows combines them into one call.

Example: Shell pipelines like cat file | grep pattern rely on fork+exec. The shell forks, sets up a pipe between the child processes, and each child execs a different program. On Windows, achieving the same requires explicit handle inheritance and explicit pipe setup in the CreateProcess call's STARTUPINFO structure.

Thread Implementation Differences

Linux clone() Windows ETHREAD 1:1 model

Both modern Linux and Windows use a 1:1 threading model, but the internal implementation differs significantly.

Linux does not have a separate "thread" abstraction in the kernel. Instead, it has clone(), a generalized system call that creates a new task. The flags you pass to clone() determine what the new task shares with the parent: CLONE_VM shares the address space, CLONE_FILES shares file descriptors, CLONE_SIGHAND shares signal handlers. A "thread" is simply a clone() call with all sharing flags set. A "process" is a clone() call with no sharing (or a fork(), which is essentially clone() with no sharing flags). This means the Linux kernel treats processes and threads as the same kind of object (a task_struct), just with different sharing configurations.

Windows has a distinct object hierarchy. A process is represented by an EPROCESS kernel structure. Each thread within it is represented by an ETHREAD structure. Processes and threads are fundamentally different object types. The Windows scheduler dispatches threads, not processes. Each thread has its own Thread Environment Block (TEB), and the process has a Process Environment Block (PEB). This explicit separation makes certain operations (like suspending all threads in a process) straightforward in Windows.

  task_struct (TID 100)                  EPROCESS (PID 200)
    flags: CLONE_VM | CLONE_FILES          ├── ETHREAD (TID 201)
                                           ├── ETHREAD (TID 202)
  task_struct (TID 101)                    └── ETHREAD (TID 203)
    flags: CLONE_VM | CLONE_FILES

Linux unifies processes and threads as task_structs. Windows separates them into EPROCESS and ETHREAD.

A notable Windows-specific mechanism is ALPC (Advanced Local Procedure Call), an internal kernel mechanism used heavily by Windows services and the window manager. It is roughly analogous to a high-performance local RPC. On the Linux side, D-Bus serves a similar role for desktop service coordination, though it runs in user space over Unix sockets rather than being a kernel primitive.

Example: On Windows, named pipes are the idiomatic way for a service to expose a local IPC endpoint (e.g., Docker Engine on Windows listens on \\.\pipe\docker_engine). On Linux, the same role is typically served by a Unix domain socket (e.g., /var/run/docker.sock). Both achieve fast local communication, but the API surface and naming conventions differ completely.

Key Takeaways

  1. A process is the unit of isolation. Use multiple processes when crashes must not propagate.
  2. A kernel thread is the unit of CPU scheduling. Use threads when tasks share data and need parallelism.
  3. Lightweight threads (goroutines, virtual threads, Erlang processes) let you scale to millions of concurrent tasks without millions of kernel objects.
  4. IPC is required between processes because they don't share memory. Shared memory is fastest but requires careful synchronization.
  5. Linux unifies processes and threads via clone(). Windows separates them into EPROCESS and ETHREAD. Both use a 1:1 kernel threading model today.
  6. Linux creates processes with fork+exec (composable, COW-optimized). Windows uses CreateProcess (explicit, single-step).
  7. The right choice depends on the workload: isolation needs, concurrency level, latency sensitivity, and the language runtime's capabilities.
