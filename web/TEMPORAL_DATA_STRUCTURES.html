Temporal Data Structures

Unit 1 — Persistent data structures, retroactive data structures, and the pointer machine model. 5 lectures.

1. Overview & Motivation

What are temporal data structures?

A temporal data structure is one that incorporates time as a first-class dimension. Instead of a single mutable state, the structure maintains information about its state across multiple points in time. This unlocks two powerful paradigms:

Persistence

Every version of the structure is preserved. You can query (and sometimes modify) any past version without destroying the current one.

Retroactivity

You can go back in time and insert, delete, or modify an operation that was performed in the past — and all subsequent states update accordingly.

Why study them?

  • Version control for data: Git-like semantics for any data structure — branch, merge, query old snapshots.
  • Computational geometry: Planar point location reduces to persistent balanced BSTs (Sarnak & Tarjan, 1986).
  • Functional programming: Immutable data structures in Haskell, Clojure, Scala are inherently persistent.
  • Error correction: Retroactive structures let you fix a bad input in the past and propagate corrections forward.
  • Undo/redo: Editors, databases, and simulations all benefit from temporal state management.

Levels of persistence

LevelQuery past?Modify past?Merge versions?
EphemeralNoNoNo
Partially persistentYes (any version)Only latestNo
Fully persistentYes (any version)Yes (any version)No
Confluently persistentYesYesYes
Key insight: Partial persistence implies a linear version history. Full persistence allows a tree of versions. Confluent persistence allows a DAG.

2. The Pointer Machine Model

Definition

A pointer machine is an abstract computational model where the storage is a directed graph of nodes connected by labeled pointers. Unlike a RAM model, pointer machines have no arithmetic on addresses — computation proceeds only by following pointers, comparing nodes, and creating or redirecting edges.

Formal definition: A pointer machine consists of:
  • A fixed alphabet Σ of labels
  • A mutable directed graph G where each node has exactly one outgoing edge per label
  • An "active" (start) node
  • Instructions: new w, set w to v, if v = w then goto z

Why does this matter for persistence?

The persistence techniques of Driscoll, Sarnak, Sleator, and Tarjan (DSST, 1986) assume the pointer machine model with bounded in-degree d. This assumption is critical because:

  • Each node has at most d incoming pointers (where d is a known constant).
  • When a node is copied (for path copying), only d parent pointers need to be updated.
  • This bounds the cascading cost of modifications, enabling O(1) amortized overhead.

Variants

ModelKey PropertyAuthor
SMM (Storage Modification Machine)Directed graph, labeled edges, mutable via setSchönhage (1980)
KU-Machine (Kolmogorov–Uspenskii)Undirected graph — pointers are invertibleKolmogorov & Uspenskii (1958)

Practical note: Real languages like C/C++ allow pointer arithmetic (RAM model), but many persistence techniques still work because we can simulate bounded in-degree by maintaining back-pointer lists.

3. Partial Persistence

Definition

A data structure is partially persistent if:

  • All past versions can be queried (read-only access).
  • Only the latest version can be modified (updated).

This creates a linear chain of versions: v₀ → v₁ → v₂ → … → vₙ (the current version).

Version numbering

Each modification increments a global version counter. Every field change is tagged with this version number. To answer a query at version t, we look up the value of each field that was current at time t.

Version:   v₀ ──→ v₁ ──→ v₂ ──→ v₃ ──→ v₄  (current)
                    │              │
                  query          query
              (read-only)    (read-only)

Only v₄ can be modified (write).

Linear version chain in partial persistence

Complexity goal

The DSST result (1986) achieves partial persistence with:

OperationOverhead
Access (any version)O(log m) worst-case, where m = total modifications
Modification (latest only)O(1) amortized time and space

4. Full Persistence

Definition

A data structure is fully persistent if:

  • All past versions can be queried.
  • Any version can be modified — not just the latest.

Modifying version vₖ creates a new version vₖ' that branches off from vₖ. The version history forms a tree.

         v₀ ──→ v₁ ──→ v₂ ──→ v₃
                  │
                  └──→ v₁' ──→ v₁''
                         │
                         └──→ v₁'''

Each node can be both queried and used as
a base for new modifications (branching).

Tree-shaped version history in full persistence

Complexity goal

DSST (1986) achieve full persistence with:

OperationOverhead
Access (any version)O(log m) worst-case
Modification (any version)O(1) amortized time and space
Version ordering: In full persistence, versions are ordered by a version tree rather than a linear sequence. Navigating this tree requires an order-maintenance data structure that supports O(1) comparisons — typically Dietz & Sleator's structure.

5. Confluent Persistence

Definition

A data structure is confluently persistent if, in addition to full persistence, there exists a merge (meld) operation that combines two versions into a new version — and the result is itself fully persistent.

      v₀ ──→ v₁ ──→ v₂
                       ╲
                        ╲──→ v₅ (merged)
                       ╱
      v₀ ──→ v₃ ──→ v₄
                         
The version history becomes a DAG.

DAG-shaped version history in confluent persistence

Why is this hard?

  • Merging two trees can produce exponential blowup in shared structure if not handled carefully.
  • Frandsen et al. (2001) showed that confluent persistence on linked data structures is harder than partial or full persistence.
  • Confluently persistent sets and maps (Liljenzin, 2013) use treaps with expected O(log n) merge.
Real-world analogy: Git branches are confluently persistent — you can branch, modify independently, and merge. The DAG of commits mirrors confluent persistence exactly.

6. Techniques: Fat Nodes

Idea

Record all changes to a node inside the node itself, without erasing old values. Each node becomes "fat" — it stores its original fields plus an arbitrary number of (version, field, value) triples.

How it works

  1. Each node stores its original field values (version 0).
  2. When a field is modified at version t, a new entry (t, field_name, new_value) is appended to the node's modification list.
  3. To read a field at version t, find the latest modification with timestamp ≤ t.
Node X:
┌───────────────────────────────────────┐
│ Original: left=A, right=B, key=5      │
│ Modifications:                        │
│   (v₃, left, C)     ← changed at v₃  │
│   (v₇, key, 8)      ← changed at v₇  │
│   (v₁₂, right, D)   ← changed at v₁₂ │
└───────────────────────────────────────┘

Query at v₅:  left=C (from v₃), right=B (original), key=5 (original)
Query at v₁₀: left=C (from v₃), right=B (original), key=8 (from v₇)

Fat node storing a history of modifications

Complexity

OperationTimeSpace
ModificationO(1)O(1) per modification
Access at version tO(log m) per node traversed—

The O(log m) slowdown comes from binary searching the modification list at each node to find the right version. This can be improved to O(log log m) using van Emde Boas trees.

Limitation

Fat nodes alone provide only partial persistence. For full persistence, the modification history at each node is no longer linear, and searching becomes more complex.

7. Techniques: Path Copying

Idea

When modifying a node, copy all nodes on the path from the root to that node. The old root points to the old version; the new root points to the new version. Unmodified subtrees are shared.

Original tree (v₀):           After insert 3 (v₁):

       5                    5  (copy)        5
      / \                  / \              / \
     3   8                2   8            3   8
    / \                  / \              / \
   2   4                1   3 (copy)    2   4
  /                        / \         /
 1                        2   4       1
                         /
                        1

v₀ root ──→ original 5     (still valid, immutable)
v₁ root ──→ copied 5       (shares subtree rooted at 8)

Path copying: only nodes on the root-to-modification path are copied

Complexity

OperationTimeSpace
ModificationO(depth) = O(log n) for balanced treesO(log n) new nodes
Access at version tO(log m) to find the right root, then O(log n) traversal—

Advantage

Path copying naturally supports full persistence — any old root can be the starting point for a new modification, creating a branch. It's the technique used in functional programming languages like Haskell and Clojure.

Disadvantage

Space can be expensive: O(log n) per modification for trees, but O(n) for a linked list (the entire "path" from head to tail must be copied).

8. The DSST Combination

The key result

Driscoll, Sarnak, Sleator, and Tarjan (1986) combined fat nodes and path copying to achieve the best of both worlds:

DSST Theorem: Any linked data structure with bounded in-degree d can be made partially or fully persistent with O(1) amortized time and space overhead per modification, and O(log m) access slowdown.

How it works

  1. Modification box: Each node stores one extra "slot" (the modification box) that can hold a single change — a field name, new value, and version stamp.
  2. If the box is empty: The modification is stored directly in the box. Cost: O(1).
  3. If the box is full: The node is copied (using the latest values), and the modification is applied to the copy. The copy has an empty modification box. Parent pointers are then cascaded (like path copying), which may trigger further copies.
Node X (modification box empty):
┌──────────────────────────────────┐
│ left=A, right=B, key=5          │
│ mod box: [ empty ]              │
└──────────────────────────────────┘
     │  Modify left→C at v₃
     ▼
┌──────────────────────────────────┐
│ left=A, right=B, key=5          │
│ mod box: [v₃, left, C]          │
└──────────────────────────────────┘
     │  Modify key→8 at v₇ (box full → copy)
     ▼
Node X' (copy, using latest values):
┌──────────────────────────────────┐
│ left=C, right=B, key=8          │
│ mod box: [ empty ]              │
└──────────────────────────────────┘
  + cascade: update X's parent to point to X'

DSST modification box mechanism

Amortized analysis

The amortized cost is proven using a potential function:

  • Φ(T) = number of full live nodes (reachable nodes whose modification box is full).
  • Each copy costs O(1) actual work but decreases the potential by 1 (a full live node is replaced by an empty live node).
  • Filling a modification box increases the potential by 1.
  • Net amortized cost per modification: O(1).
Intuition: Copies are "paid for" by the potential released when full nodes are replaced. The modification box acts as a buffer — most modifications are absorbed without copying.

9. Persistent BST Example

Path-copying persistent BST

The simplest practical persistent data structure: a balanced BST where each insert/delete creates a new root by copying the path from root to the modified leaf.

pythonclass Node:
    def __init__(self, key, left=None, right=None):
        self.key = key
        self.left = left
        self.right = right

def insert(root, key):
    """Returns a NEW root — old root is unchanged."""
    if root is None:
        return Node(key)
    if key < root.key:
        # Copy current node, recurse left
        return Node(root.key, insert(root.left, key), root.right)
    elif key > root.key:
        # Copy current node, recurse right
        return Node(root.key, root.left, insert(root.right, key))
    else:
        return root  # Duplicate key, no change

def search(root, key):
    if root is None:
        return False
    if key == root.key:
        return True
    if key < root.key:
        return search(root.left, key)
    return search(root.right, key)

# Usage — each insert returns a new version
versions = []
v0 = None
v1 = insert(v0, 5)    # versions: [5]
v2 = insert(v1, 3)    # versions: [5, 3]
v3 = insert(v2, 8)    # versions: [5, 3, 8]
v4 = insert(v3, 1)    # versions: [5, 3, 8, 1]

# Query old versions — they're still intact!
print(search(v2, 8))   # False — 8 wasn't in v2
print(search(v3, 8))   # True  — 8 was added in v3
print(search(v4, 3))   # True  — 3 is still in v4

Persistent segment tree (competitive programming)

One of the most interview-relevant persistent structures. Each point update creates O(log n) new nodes via path copying.

pythonclass SegNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

def build(lo, hi):
    if lo == hi:
        return SegNode()
    mid = (lo + hi) // 2
    return SegNode(0, build(lo, mid), build(mid + 1, hi))

def update(node, lo, hi, idx, delta):
    """Returns a NEW root with arr[idx] += delta."""
    if lo == hi:
        return SegNode(node.val + delta)
    mid = (lo + hi) // 2
    if idx <= mid:
        return SegNode(
            node.val + delta,
            update(node.left, lo, mid, idx, delta),
            node.right   # shared — no copy
        )
    else:
        return SegNode(
            node.val + delta,
            node.left,   # shared — no copy
            update(node.right, mid + 1, hi, idx, delta)
        )

def query(node, lo, hi, l, r):
    """Sum query on range [l, r]."""
    if node is None or lo > r or hi < l:
        return 0
    if l <= lo and hi <= r:
        return node.val
    mid = (lo + hi) // 2
    return query(node.left, lo, mid, l, r) + \
           query(node.right, mid + 1, hi, l, r)
Interview tip: Persistent segment trees are used to answer "kth smallest in a range" queries. Store sorted-order insertion versions and use binary search across two version roots.

10. Retroactive Data Structures

What is retroactivity?

A retroactive data structure supports modifications to the sequence of operations that have been performed on it. You can insert, delete, or update an operation at any point in the past, and the structure reflects what would have happened if the modified history had been the original.

How it differs from persistence

Persistence

Preserves old versions as immutable snapshots. Modifying a version creates a new branch — old versions are never altered.

Retroactivity

Modifies the timeline itself. Inserting an operation at time t₃ changes the state at t₃, t₄, t₅, … — a cascading ripple effect.

Formal setup

Given a sequence of operations U = [ut₁, ut₂, …, utₘ] ordered by time:

  • Insert(t, op) — insert operation op at time t in the history.
  • Delete(t) — remove the operation that was at time t.

After either operation, the current state is recomputed as if the modified sequence had always been the history.

Key constraint: There is no general efficient technique to make any data structure retroactive. Demaine, Iacono, and Langerman (2007) proved that efficient retroactivity must exploit the specific structure's properties.

11. Partial Retroactivity

Definition

A data structure is partially retroactive if:

  • You can Insert(t, op) and Delete(t) operations in the past.
  • You can only query the current state (the state after all operations in the modified timeline).

Example: partially retroactive stack

Timeline:  push(3)  push(7)  pop()  push(5)
State:        [3]   [3,7]    [3]    [3,5]    ← current state: [3,5]

Insert(t₁.₅, push(9)):
Timeline:  push(3)  push(9)  push(7)  pop()  push(5)
State:        [3]   [3,9]   [3,9,7]  [3,9]   [3,9,5]  ← new current: [3,9,5]

A retroactive insert ripples through all subsequent states

Running times

Let m = total operations, r = operations before the retroactive change, n = max elements at any time:

Data StructurePartial Retroactivity Cost
Queue / DequeO(log m)
Priority QueueO(√m · log m)
Union-FindO(m · α(m))

12. Full Retroactivity

Definition

A data structure is fully retroactive if in addition to retroactive insert/delete, you can also query at any past time:

  • Insert(t, op) — add operation at time t.
  • Delete(t) — remove operation at time t.
  • Query(t, q) — answer query q as of time t (in the modified timeline).

Difficulty

Full retroactivity is generally harder than partial because every retroactive change can affect the answer at any time point, not just the present. The cascading effect must be tracked at all intermediate states.

Think of it this way: Partial retroactivity = "change the past, see the present." Full retroactivity = "change the past, inspect any moment in the modified timeline."

13. Retroactive Priority Queue

The canonical example

The priority queue is the most studied retroactive structure. Operations are insert(x) and delete-min(), and we want to retroactively add or remove these at any past time.

The geometric view

Map the timeline onto the x-axis. For each element, draw a horizontal line segment from its insertion time to its deletion time. The elements currently in the queue at time t are exactly the segments that cross the vertical line at t.

value
  │
  │   ────────── 7 ──────────
  │        ─────── 5 ─────
  │   ── 3 ──
  │              ───── 2 ─────────
  │
  └──────────────────────────────→ time
       t₁    t₂    t₃    t₄

At time t₂: elements in queue = {7, 5, 3}
delete-min at t₂ removes 3 (shortest value segment)

Geometric view: each element is a time interval

Key insight — bridges

A retroactive insert or delete changes which element is the minimum at some time range. The "bridge" structure tracks, for each time interval, whether the set of present elements has changed — enabling efficient updates in O(√m · log m) for partial retroactivity.

15. Applications

Planar point location

Given a planar subdivision (e.g., a map of polygonal regions), answer: "which region contains point p?"

Solution (Sarnak & Tarjan, 1986): Sweep a vertical line from left to right. Maintain a persistent balanced BST of line segments ordered by y-coordinate. At each event point (segment start/end), update the BST. To query point p = (x, y), find the BST version at time x and search for y.

Result: O(n log n) preprocessing, O(log n) query, O(n log n) space (using path copying).

Persistent segment trees in competitive programming

  • Kth smallest in range [l, r]: Build persistent segment tree on sorted values. Version i includes elements a[0..i]. Answer = walk two version roots (version r minus version l-1).
  • Online queries on arrays: Each update creates a new version; past queries use old roots.

Functional data structures

  • Haskell, Clojure, Scala: All standard library collections are persistent. Lists use structural sharing; maps use persistent red-black trees or HAMTs.
  • Redux (JavaScript): State management via immutable persistent data, enabling time-travel debugging.

Error correction & recovery

  • A sensor network produces erroneous data. Retroactively remove the bad readings and recompute the system state.
  • A database transaction is found to be incorrect hours later. Retroactively undo it and replay subsequent transactions.
