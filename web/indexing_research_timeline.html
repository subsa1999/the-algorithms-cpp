Index Families Explained

The sections below are designed to be correct, example driven, and diagram driven. They focus on what each family of indexes is trying to optimize, what the core data structure looks like, and where it wins or loses in real systems.

1. Database Indexes: B-trees, B+ trees, GiST, LSM-trees, and Bitmap Indexes

Database Ordered access Range queries Transactional systems

Database indexing begins with a practical constraint rather than a purely mathematical one: storage is slow, memory is limited, and random I/O is expensive. A database index is therefore not just a way to order keys. It is a strategy for reducing the number of storage pages that must be touched in order to answer a query. That single design pressure explains why B-trees became dominant. A binary search tree has low comparison complexity in theory, but if each node lives on a different disk page, then a deep tree becomes expensive. A B-tree uses high fan-out instead. One internal node can hold many separator keys and many child pointers, so a search descends only a few levels before reaching the target page.

The classic example is an employee table indexed by employee ID. Instead of comparing against one key per node, the root page may contain dozens or hundreds of keys, each carving the key space into broad intervals. The search path becomes short, stable, and page oriented. B+ trees refine the same idea by keeping data records only in the leaves, which makes internal nodes denser and therefore even shallower. This is why production relational systems typically prefer B+ tree variants over textbook B-trees. Internal pages are optimized for navigation. Leaf pages are optimized for storage and sequential scans. Once a query reaches the first leaf in a range, it can follow leaf-to-leaf links to continue scanning with minimal overhead. 

                 [ 20 | 40 | 70 ]
                /       |      |       \
      [ 1..19 ]  [20..39] [40..69] [70..99]

Leaf pages in a B+ tree:

[ 01 05 11 18 ] -> [ 20 22 31 38 ] -> [ 40 44 51 67 ] -> [ 70 72 83 95 ]

Point lookup:
  search 44 -> root interval 40..69 -> leaf [40 44 51 67]

Range query:
  search 22..67 -> first matching leaf, then scan linked leaves forward
Example: Suppose a payroll table stores 50 million rows and the query is “find all salaries for employee IDs from 400000 to 420000.” A hash index is good at exact lookup, but it does not preserve order. A B+ tree can find the first matching leaf and then perform an efficient range scan across adjacent leaves.

GiST generalizes the idea of a balanced search tree. Instead of saying that every node must behave like an ordered set of scalar keys, GiST says the system should provide a framework in which different notions of “consistency,” “union,” “penalty,” and “split” can be plugged in. This matters because many real data types are not simple integers or strings. Geometric objects, document signatures, interval values, and custom domain-specific objects often need specialized search semantics. GiST does not replace B+ trees conceptually; it provides a general infrastructure for implementing index families that can still live inside a database engine with concurrency control and recovery.

LSM-trees attack a different bottleneck. B-tree families are excellent when a system needs balanced lookup and range behavior with moderate writes. But if the workload is dominated by heavy inserts and updates, random page rewrites become costly. LSM-trees buffer writes in memory, flush them sequentially to disk as sorted runs, and later merge those runs in the background. This reduces write amplification from small random updates at the page level, trading it for compaction work and more complicated reads. In systems like log storage, time-series ingestion, or write-heavy key-value stores, this trade is often worth it. 

LSM-tree write path:

Incoming writes -> Memtable (RAM) -> Flush -> Sorted Run L0 -> Merge/Compaction -> L1 -> L2

Read path:
  1. Check Memtable
  2. Check newest sorted runs first
  3. Use Bloom filters to skip runs that cannot contain the key

Strength:
  very fast sustained writes

Cost:
  reads may search multiple levels and background compaction must be managed

Bitmap indexes serve yet another regime: analytical queries over large tables with repeated low-cardinality attributes. Imagine a retail warehouse with columns such as region, channel, returned flag, and weekday. For each distinct value, a bitmap marks which rows match. Complex conditions then become bitwise operations. If the query asks for all rows where region is East, channel is Online, and returned flag is false, the engine can AND three bitmaps and identify qualifying rows very quickly. The structure is especially powerful in read-mostly systems because compression works well and logical operations are fast on compressed or word-aligned bitmaps.

Example: In a fact table with 200 million sales records, a bitmap index on the boolean field “promotion_applied” is tiny and extremely selective for analytical reporting. A B-tree can answer the same predicate, but the bitmap can compose naturally with other bitmaps such as region, store type, or quarter.

The main lesson is that “database index” is not one thing. It is a family of engineering answers to different workload shapes. B+ trees dominate mixed read and range workloads. GiST helps when the key type is not naturally one-dimensional. LSM-trees dominate when writes are the bottleneck. Bitmap indexes dominate where predicates are compositional and data is warehouse-like. Choosing correctly is less about abstract asymptotic complexity and more about page layout, update rate, range behavior, compression, and concurrency.

2. Spatial Indexes: R-trees and the Problem of Searching Geometry

Spatial Multidimensional GIS Nearest neighbor

Spatial indexing starts from a problem that breaks ordinary ordered indexes. If the data is a set of rectangles, polygons, road segments, or latitude-longitude points, then there is no single scalar ordering that preserves all proximity relationships. Sorting by x-coordinate alone loses information about y. Sorting by area is even worse. The core challenge is to organize multidimensional objects so that queries can eliminate large parts of the search space quickly without examining every object directly.

R-trees solve this by grouping nearby objects and summarizing each group with a minimum bounding rectangle, often abbreviated MBR. Each internal node stores a set of bounding rectangles and child pointers. A query rectangle, such as “all restaurants within this area,” traverses only children whose MBR intersects the query box. This is a filter-and-refine approach. Internal rectangles are coarse approximations. If a child rectangle does not overlap the query, none of its descendants can possibly match. If it does overlap, the search continues and later refines the result against the true objects. 

Query box Q intersects only selected regions:

Root:
  [R1] [R2] [R3]

If Q intersects R2 and R3, search only those subtrees.

2D intuition:

+-----------------------------+
| R1      | R2 overlaps Q     |
|         |   +------+        |
|         |   |  Q   |        |
|         |   +------+        |
|-----------------------------|
| R3 overlaps Q partially     |
+-----------------------------+

R1 is skipped entirely.

This idea is simple, but the quality of the tree depends heavily on how rectangles are grouped. If sibling rectangles overlap too much or cover large empty regions, the query must descend into many branches, and performance degrades. That is why R-tree research focuses so intensely on split heuristics and bulk loading. The original R-tree introduced dynamic insertion and page splitting, but later work such as the R*-tree improved practical behavior by minimizing overlap and sometimes reinserting entries instead of splitting immediately. The goal is not merely balance. It is balance plus low overlap plus compact covering regions.

Example: Consider a map application storing every building footprint in a city. A viewport query should return only objects intersecting the user’s current screen. Without a spatial index, the system would test every building polygon. With an R-tree, it first tests query overlap against a handful of directory rectangles, descends only into matching subtrees, and touches a much smaller candidate set.

Spatial indexes also support nearest-neighbor search. Here the query is not “which objects intersect this box?” but rather “which gas station is closest to my current position?” The R-tree can prioritize nodes by minimum possible distance from the query point to each bounding rectangle. A priority queue explores the most promising rectangles first. Whenever the algorithm reaches a true object whose distance is smaller than the remaining candidate rectangles, it can prune aggressively. This makes R-tree families useful not just for window queries but for route planning, map exploration, clustering, and location-based services.

One subtle but important detail is that spatial indexes usually index simplified geometry, not the full object. A polygon may have thousands of vertices, but the directory structure stores only bounding rectangles. That means the tree acts as a fast filter. After candidate objects are found, the system may still need an exact geometric predicate such as point-in-polygon or polygon intersection. This is why spatial indexing often appears together with the phrase “filter and refine.” The index gets the candidate set small enough that exact geometry becomes affordable. 

Spatial query pipeline:

1. Index filter:
   test query box against MBRs

2. Candidate set:
   buildings A, F, H survive

3. Exact refinement:
   run full polygon intersection only on A, F, H

This avoids expensive exact geometry on the entire dataset.

Spatial indexes are therefore best understood as approximate directory structures over geometric data. Their performance depends less on raw tree height and more on geometric quality: overlap, dead space, and locality. This makes them fundamentally different from scalar ordered indexes. A B+ tree search usually follows one path. An R-tree query may follow several paths because geometric regions overlap. Good spatial indexing is about minimizing how often that branching explosion happens.

In practice, this family matters anywhere the data lives in two or more dimensions: geographic information systems, computer-aided design, robotics, game engines, computer vision, and geospatial analytics. The R-tree and R*-tree became foundational not because they are perfect in theory, but because they capture the right compromise between dynamic updates, disk awareness, and geometric pruning.

3. Text and String Indexes: Suffix Trees, Suffix Arrays, and Inverted Indexes

Text Indexing Pattern matching Search engines Bioinformatics

Text indexing divides naturally into two major problem settings. The first is substring search over a single long text or a collection treated as raw strings. The second is term-based document retrieval, where the query is expressed in words rather than arbitrary substrings. Suffix trees and suffix arrays belong to the first setting. Inverted indexes dominate the second. They solve related but distinct problems, and it is a mistake to treat them as interchangeable.

A suffix tree stores every suffix of a string in a compressed trie-like structure. This sounds extravagant, but it gives extremely strong search behavior. If the text is banana$, then the suffixes are banana$, anana$, nana$, ana$, na$, a$, and $. By compressing common prefixes on edges, the structure can answer substring existence queries in time proportional to the pattern length. That was a major breakthrough because it turned many string problems into direct tree traversals rather than repeated rescans of the text. 

Text: banana$

Sorted suffixes:
  $        -> 7
  a$       -> 6
  ana$     -> 4
  anana$   -> 2
  banana$  -> 1
  na$      -> 5
  nana$    -> 3

Substring search for "ana":
  suffix tree path for a -> n -> a succeeds
  so "ana" occurs in the text

The weakness of suffix trees is space. They are powerful, but they are not cheap. Real implementations often require several times the size of the original text. That cost motivated suffix arrays. A suffix array stores the starting positions of suffixes in sorted order, not an explicit tree of nodes and edges. Search then becomes a binary search over the sorted suffixes. With extra support data such as LCP arrays, suffix arrays recover much of the functionality of suffix trees while using substantially less space and often better cache behavior.

Example: Suppose the text is a chromosome or a very large log file, and the query is “find every occurrence of ACTGGA” or “find every occurrence of ERROR42.” A suffix tree or suffix array treats the text as a character stream and supports arbitrary substring search. That is different from a search engine query like “distributed systems consensus,” which is usually word based and document based.

Inverted indexes answer the search engine style problem. They map each term to the documents, and often positions, in which the term appears. If the collection contains the documents “graph algorithms,” “distributed graph processing,” and “database indexing,” then the posting list for graph might be [doc1, doc2]. A query containing multiple terms becomes posting list intersection, union, ranking, and positional filtering. This structure is vastly more practical than suffix-based structures for web-scale document retrieval because the natural unit of search is often the token rather than the arbitrary substring. 

Inverted index example:

graph      -> [doc1, doc2]
distributed-> [doc2]
indexing   -> [doc3]

Query: "graph AND distributed"
  intersect [doc1, doc2] with [doc2]
  result -> [doc2]

Positional index example:
consensus -> [(doc7, positions 3, 14), (doc9, position 8)]

The distinction matters because the semantics differ. A suffix array can answer substring queries such as “find all occurrences of ana in banana.” An inverted index, unless built positionally over characters or k-grams, does not naturally support that query. Conversely, an inverted index is naturally aligned with ranked retrieval over documents, TF-IDF style scoring, phrase queries, Boolean logic, and posting list compression. Search engines therefore center on inverted indexes, sometimes augmented by forward indexes, document statistics, positional data, or specialized sub-indexes.

Another important point is that text indexing is not only for natural language. Bioinformatics relies heavily on string indexes because DNA and proteins are strings over small alphabets. Log analytics uses them for pattern localization. Security analytics uses them for signature search. Plagiarism detection, corpus mining, autocomplete, and computational linguistics all depend on some form of efficient string or token index.

Suffix trees represent the most expressive classical structure. Suffix arrays provide a cleaner and often more practical formulation. Inverted indexes dominate document retrieval. The correct mental model is not “which one is best?” but “what exactly is being searched: arbitrary substrings, or document terms?” Once that question is answered, the structure choice becomes much clearer.

4. Compressed Indexes: FM-indexes and Compressed Suffix Arrays

Compressed Indexing Self-indexing Entropy-aware Large texts

Compressed indexing is one of the most elegant ideas in modern algorithms because it tries to do two things at once that once seemed to be in conflict: compress the data and still allow fast search over it. Classical full-text indexes such as suffix trees and suffix arrays support strong query capabilities, but their space cost can be large. When the dataset is enormous, such as whole genomes or web-scale corpora, index size becomes a first-order systems problem. A compressed index attacks that problem directly.

The FM-index is built around the Burrows-Wheeler Transform, or BWT. The transform rearranges the text so that similar contexts cluster together, making the result highly compressible. The remarkable step is that this transformed representation is not merely compressed data. With the right auxiliary structures, it becomes searchable. In other words, the transformed text plus rank information and a small amount of navigation data acts as a self-index: the text is compressed, but the index can still count and locate pattern occurrences. 

High-level FM-index idea:

Text T -> Burrows-Wheeler Transform -> compressed representation

Then add:
  C[c]   = how many characters are lexicographically smaller than c
  Occ(c, i) = how many times c appears up to position i

Backward search:
  process pattern from right to left
  shrink the suffix interval each step

Result:
  find all suffixes beginning with the pattern without storing the full suffix tree

To see why this matters, consider the pattern bra inside abracadabra$. A suffix array would search among sorted suffixes. The FM-index instead keeps an interval of candidate suffixes and updates that interval one character at a time, from the end of the pattern toward the front. The interval narrows as the algorithm uses rank information to jump through the transformed text. This is conceptually different from explicit tree traversal, but it achieves the same core purpose: identify the suffixes that begin with the query pattern.

Example: Modern DNA read aligners often rely on FM-index style techniques because they need exact or near-exact pattern search over massive reference genomes. A human genome is large enough that a space-hungry classical index can become a serious memory burden. A compressed self-index dramatically changes what is practical.

Compressed suffix arrays pursue the same goal from a slightly different angle. Instead of storing the full suffix array explicitly, they compress it in a way that still supports navigation and pattern matching. The theoretical insight is deep: the index size can be tied to the empirical entropy of the text rather than to a blunt worst-case bound like storing every suffix position naively. This means the structure takes advantage of regularity in the source text. Highly repetitive texts become especially attractive targets.

The phrase self-index is important. In a self-index, the index can substitute for the original text rather than merely point into it. That makes compressed indexes conceptually stronger than “compressed plus an external index.” The compressed representation and the search structure are fused. This fusion is why the FM-index became such a landmark result. It changed the framing of the problem from “store an index over the text” to “let the compressed text itself become the index.” 

Classical approach:
  original text + separate index

Compressed self-index approach:
  compressed searchable representation

Space trade-off:
  suffix tree    -> very powerful, high space
  suffix array   -> simpler, lower space
  FM-index / CSA -> searchable while near compression bounds

The main trade-offs are predictable. Compressed indexes usually reduce memory footprint dramatically, but their navigation is more intricate than direct pointer chasing in a tree or direct array access in a suffix array. Constant factors matter. Locality matters. Implementation quality matters. In-memory versus external-memory behavior matters. This is why compressed indexing research includes not only theory papers but also experimental work on construction, navigation, and practical engineering.

These indexes are especially compelling in domains where the collection is huge, patterns are frequent, and memory is precious. Genomics is the canonical example. Large archival text collections are another. Compressed indexes also matter because they pushed a broader algorithmic insight into the mainstream: information-theoretic structure should influence data structure design. A good index should not merely answer queries fast. It should exploit redundancy in the data itself.

The correct way to think about this family is as the convergence of full-text indexing and compression. Classical full-text indexing asked how to search quickly. Compression asked how to store compactly. FM-indexes and compressed suffix arrays show that, with the right mathematics, the same structure can do both.
